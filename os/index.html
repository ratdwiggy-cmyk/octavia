OPERATING SYSTEM

The structure behind every piece of work Octavia publishes

Octavia isn't a portfolio of finished-looking things — it's a running record of one repeated process, applied to different problems. This page is the index of that process: six sections that hold the work at each stage, and one pipeline that every piece of work moves through on its way to being published. For how aligned people can engage with this work directly, see Collaborate.

The pipeline

Every entry in the OS — regardless of which section it lives in — is understood in terms of these six stages. An entry doesn't need to have reached "Result" to be published; its stage badge just says where it honestly is right now.

  1. 01 Problem What's broken, slow, or missing — stated concretely, for someone in particular.
  2. 02 Research What's already known: prior art, constraints, and the approaches worth trying first.
  3. 03 Prototype The smallest version that could possibly work, built to be tested rather than shipped.
  4. 04 Test Running the prototype against a real case and recording what actually happened.
  5. 05 Improvement What the test revealed, and the specific change made in response to it.
  6. 06 Result The outcome in plain terms, plus what it changes about the next problem.

ADDING NEW WORK

Every section uses the same reusable template

HOW THIS OS STAYS EASY TO EXTEND

Each section page (Projects, Research, Experiments, Tools, Frameworks) has a dashed "+ New entry" card at the end of its grid — copy that card's HTML to add a new item to the list. For full entries, duplicate entry-template.html, rename it, fill in the six pipeline stages that apply, and point the new card's link at the renamed file. The Learning Log uses a lighter dated-row template instead of the full pipeline, since most log notes don't need all six stages.

  1. Open the section page and copy the <!-- TEMPLATE START --> block.
  2. Duplicate entry-template.html as your-entry-name.html (skip this for Tools/Frameworks if a one-line catalog entry is enough).
  3. Fill in the bracketed placeholders and update the stage badge to match where the work actually is.
  4. Commit and push — GitHub Pages rebuilds automatically.

See the entry template directly, or read README.md for the full walkthrough.
